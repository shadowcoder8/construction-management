## 2024-04-25 - [Semantic HTML for Dashboard Cards]
**Learning:** Found an accessibility issue pattern specific to this app where dashboard navigation cards were created using `<div>` tags with `onclick` handlers, relying on JavaScript for navigation. This breaks native keyboard navigation, screen reader support, and standard browser actions (like opening in a new tab). Also, using `alt` text identical to visible text in cards causes redundant screen reader announcements.
**Action:** Always prefer semantic HTML elements (like `<a>` for navigation links) over `<div>`s with JavaScript handlers. For images in links where the link text already describes the destination, use `alt=""` for the image. Ensure interactive elements have distinct focus states using `:focus-visible`.

## 2024-06-05 - [Accessibility for Placeholder-Driven Forms]
**Learning:** Found an accessibility issue pattern specific to this app where input forms heavily rely on `placeholder` attributes instead of visible `<label>` tags. This creates accessibility barriers for screen readers, which rely on labels to communicate form field purpose. Additionally, adding `aria-label` to hidden inputs (`<input type="hidden">`) causes screen readers to incorrectly interpret them as interactive elements.
**Action:** When working with forms that use placeholders instead of visible labels to maintain visual layout, always add corresponding `aria-label` attributes to visible `input` and `select` elements. Crucially, explicitly avoid adding `aria-label` to any `hidden` inputs to prevent confusing screen reader announcements.
